- [confirmed] A February 2026 ETH Zurich/LogicStar.ai study (Gloaguen et al., arXiv:2602.11988, 'Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?') built a 138-task benchmark (AGENTbench) from niche (non-memorized) Python repos and tested Claude Sonnet 4.5, Codex GPT-5.2/GPT-5.1-mini, and Qwen Code across no-file / LLM-generated-file / human-written-file conditions: LLM-generated files reduced task success rate by ~3% on average (worse in 5 of 8 settings) and added 20-23% inference cost plus 2.45-3.92 extra agent steps per task; human-written files improved success by ~4% but still cost ~19% more; using a stronger model to generate the file did not help.
  IMPL: This is the single most important effectiveness data point for redesigning rules.md: it is direct empirical evidence that (a) AI-generated rules files can actively hurt task success, (b) even good human-written files carry a real cost tax, and (c) the study's own recommendation — 'omit LLM-generated context files entirely and limit human-written instructions to non-inferable details' — matches dev-rules' existing philosophy of hand-curated, evidence-gated rules rather than auto-generated boilerplate.

===== LITERATURE =====
- [confirmed] Prompt-format effects (Markdown/XML/JSON/YAML/TOON) on instruction adherence are real but model-dependent, and are generally dwarfed by raw model-capability differences; weaker/smaller models show far larger format sensitivity than frontier models.
  IMPL: Do not assume rules.md format choice is neutral across the tool ecosystem this repo targets (Claude, GPT, Cursor, Copilot, DeepSeek, Qwen, GLM). Markdown is the safest universal default: no format is a clear frontier-model winner, but non-Markdown structured formats (raw JSON/TOON) showed measurable accuracy costs (9-14pp) and parsing failure cascades on tool-call-heavy agentic tasks in weaker/open models.
  SRC: https://arxiv.org/abs/2411.10541; https://arxiv.org/abs/2605.29676
- [single-source] Official vendor guidance (Anthropic, OpenAI) still recommends XML tags / markdown headings for structuring prompts, but independent benchmarks find XML often underperforms YAML and costs substantially more tokens for nested/structured data.
  IMPL: Treat vendor-recommended XML tagging as a per-tool optimization, not a universal rule for a cross-tool rules.md; a table/markdown-heading structure (already dev-rules.md's approach) avoids the token overhead and open-model degradation XML showed in this benchmark. Needs a second independent source before treating as settled.
  SRC: https://www.improvingagents.com/blog/best-nested-data-format/
- [single-source] Imperative-mood instructions ('NEVER do X') carry inconsistent, register-dependent obligatory force across languages/model training distributions; declarative restatement ('X: disabled') is more stable and transfers better cross-lingually.
  IMPL: This is genuinely new evidence not in the existing base. For a rules file meant to be 'universally understood' across model families, this suggests hedging PROHIBITION-style rules with a declarative-state fallback phrasing (e.g., pairing 'Never hardcode secrets' with a factual framing like 'Hardcoded secrets: not permitted') could improve cross-model/cross-language reliability. Single paper — do not over-weight without replication.
  SRC: https://arxiv.org/abs/2603.25015
- [confirmed] Negatively-framed ('do not') instructions are empirically less reliable than positively-framed explicit instructions because the model must process the suppressed content to negate it, activating rather than suppressing it (LLM analog of ironic process theory); but the direction/magnitude of framing effects is task-dependent, not uniform.
  IMPL: Reinforces existing dev-rules.md pattern of pairing prohibitions with positive alternatives (e.g. 'Never Y' followed by 'Do X instead'). New nuance: DeFrame shows framing risk is NOT uniformly 'negative framing = always worse' — direction can flip by task/content type, so blanket positive-only rewriting isn't guaranteed safe; treat as a strong default, not an absolute law.
  SRC: https://arxiv.org/abs/2605.05391; https://arxiv.org/abs/2602.04306
- [contested] System-prompt placement is not unconditionally superior to user-message placement for persistent instruction adherence in multi-turn settings; one 2026 study found the opposite of the conventional 'system > user' assumption in its domain.
  IMPL: Flag as a caveat, not a rule change: this finding is from spoken/audio language models (SLMs), a different domain than text-based coding agents, and conflicts with the 'system prompts dominate' literature the paper itself cites (Neumann et al. 2025). Do not migrate dev-rules content out of the system/CLAUDE.md slot on this evidence alone; record as an open contradiction pending a text-domain replication.
  SRC: https://arxiv.org/abs/2512.23578
- [single-source] For long agentic sessions, event-driven/state-triggered instruction reminders injected right before a decision point outperform periodic full system-prompt re-injection, which wastes tokens on irrelevant instructions.
  IMPL: This is a harness-layer mechanism (event-triggered reminders), explicitly out-of-scope for rules.md per this repo's own CLAUDE.md 'Out of Scope for rules.md' table — but it is useful upstream evidence that the dev-skills orchestration layer (not rules.md itself) should consider re-injecting salient rules.md excerpts at decision points in long sessions rather than relying on one-time system-prompt loading.
  SRC: https://arxiv.org/abs/2603.05344
- [single-source] A rigorous controlled study (ETH Zurich / LogicStar, 138-repo AGENTBENCH + SWE-bench Lite validation) found that context files (AGENTS.md) do NOT generally improve task success rates and increase inference cost; LLM-auto-generated context files actively hurt (reduced resolve rates, agents followed file instructions literally even when counterproductive), while developer-written files showed only modest, benchmark-dependent gains.
  IMPL: Directly relevant new counter-evidence to arXiv 2604.11088's 'guardrails beat guidance' framing: even a well-designed guardrails-style file does not reliably move task success on its own.
